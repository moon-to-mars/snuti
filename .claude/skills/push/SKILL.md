# Push 단계

## 목적
feature의 모든 하위 항목 구현이 끝났을 때, 원격에 push하고 PR을 생성한 뒤 브랜치를 정리한다.
브랜치·커밋·PR은 사용자가 직접 실행한다.

## 절차

### 1. 완료 요약
이번 feature에서 작업한 하위 항목 전체와 주요 변경 파일을 정리해서 보여준다.

```
## feature 완료 요약
- feature: [feature 항목명]
- 완료된 하위 항목:
  - [x] [항목1]
  - [x] [항목2]
- 주요 변경 파일: [파일 목록]
```

### 2. PR 전 체크포인트 — 1개씩 확인
아래 항목을 **하나씩** 제시한다. 사용자가 응답하면 다음 항목으로 넘어간다.

항목마다 아래 형식으로 제시한다:
```
(N/3) [항목 내용]
1. 확인 완료
2. 문제 있어 (어떤 문제인지 말해줘)
3. 해당 없음
```

확인 항목:
1. feature 브랜치에서 작업했는가?
2. 기능 하나 = 커밋 하나 원칙 지켰는가?
3. 리팩토링을 다음 기능 전에 했는가?

"문제 있어"를 선택하면 해당 문제를 즉시 처리한 뒤 다음 항목으로 넘어간다.

### 3. checklist.md 최상위 항목 업데이트 및 커밋
해당 feature의 최상위 항목을 `- [ ]` 에서 `- [x]` 로 변경한다.

변경 후 반드시 커밋 안내를 한다:
```
git add checklist.md
git commit -m "docs: feature/[번호] 완료 체크"
```

### 4. push 안내
사용자에게 아래 명령어를 보여주고 직접 실행하도록 안내한다:

```
git push -u origin [현재 브랜치명]
```

### 5. PR 생성 안내
push 완료 후 아래 내용을 제안하고, 사용자가 GitHub에서 PR을 생성하도록 안내한다.

**PR 방향**: `feature/[이슈번호]-[기능명]` → `dev`

**PR 제목** 예시:
```
[feature/[이슈번호]-[기능명]] [feature 항목명]
```

**PR 본문**을 아래 형식으로 직접 작성해서 제안한다. 이번 feature에서 실제로 작업한 내용을 구체적으로 채운다:

```
## 작업 내용

### [카테고리1 — 예: 백엔드, 미들웨어, 디렉토리 구조 등]
- 구체적인 작업 항목 (파일명, 방식 등 포함)
- ...

### [카테고리2]
- ...

## 커밋 목록
- [커밋 메시지1]
- [커밋 메시지2]
- ...

## 관련 브랜치
feature/[이슈번호]-[기능명] → dev

Closes #[이슈번호]
```

카테고리는 작업 성격에 따라 자유롭게 나누되, 항목마다 **무엇을 왜 했는지** 알 수 있을 만큼 구체적으로 쓴다.
`Closes #[이슈번호]`는 PR 머지 시 연결된 GitHub Issue를 자동으로 닫아준다. 이슈 번호는 브랜치명에서 추출한다.

### 6. 브랜치 삭제 안내
PR 머지 후 아래 명령어로 로컬 브랜치를 삭제하도록 안내한다:

```
git branch -d feature/[이슈번호]-[기능명]
```

원격 브랜치는 GitHub PR 머지 시 "Delete branch" 버튼으로 삭제하도록 안내한다.

## 완료 조건
브랜치 삭제까지 완료되면 이 workflow가 종료된다.
