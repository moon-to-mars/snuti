# 리뷰 단계

## 목적
구현된 코드의 품질과 정확성을 검증하고 사용자의 피드백을 받는다.

## 절차

### 1. 변경사항 요약
이번 구현에서 수정/생성된 파일과 핵심 변경 내용을 간단히 정리한다.

설계 단계에서 합의했던 내용과 다르게 구현된 부분이 있다면 그 이유도 함께 언급한다.

### 2. 검증 체크포인트 — 1개씩 확인
설명이 끝나면 아래 항목을 **하나씩** 제시한다. 사용자가 각 항목을 확인하고 응답하면 다음 항목으로 넘어간다.

항목마다 아래 형식으로 제시한다:
```
(N/총개수) [항목 내용]
1. 확인 완료
2. 문제 있어 (어떤 문제인지 말해줘)
3. 해당 없음
```

**코드 확인 항목 — 항상 확인 (4개)**
1. 내가 설계한 대로 만들었는가?
2. 이해 안 되는 코드가 있는가?
3. 내가 요청하지 않은 것이 들어왔는가?
4. CLAUDE.md 금지 패턴이 포함됐는가?

**브라우저 확인 항목 — 프론트엔드 코드일 때만 확인 (4개)**
백엔드 코드(routes, services, repositories, models)라면 이 항목들은 전부 건너뛴다.
5. 의도한 레이아웃으로 보이는가?
6. 로딩 상태가 노출되는가?
7. 에러 상태가 처리됐는가?
8. 개발 환경에서만 동작하는 코드가 없는가? (fetch('/src/data/...') 같은 패턴)

"문제 있어"를 선택하면 해당 문제를 즉시 처리한 뒤 다음 항목으로 넘어간다.

### 4. CLAUDE.md 금지 패턴 업데이트 검토
모든 체크포인트를 통과하면, 이번 구현에서 나온 패턴 중 **앞으로 모든 코드에 적용해야 할 규칙**이 있는지 능동적으로 검토한다.

추가할 만한 규칙이 있으면 제안하고 사용자 동의 시 CLAUDE.md의 `### 금지 패턴 (server)` 또는 해당 섹션에 추가한다.
추가할 내용이 없으면 이 단계를 건너뛴다.

### 5. 피드백 요청
CLAUDE.md 업데이트(또는 건너뜀) 후 아래 형식으로 피드백을 요청한다.

```
모든 항목 확인 완료. 추가로 수정하고 싶은 게 있나요?
1. 수정할 부분이 있어
2. 좋아, 커밋하러 가자
```

## 완료 조건
- "수정할 부분이 있어" → workflow가 `confirm` 스킬을 발동한다
- "좋아, 커밋하러 가자" 또는 동의 표현 → workflow가 `commit` 스킬을 발동한다
