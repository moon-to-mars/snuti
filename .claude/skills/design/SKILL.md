# 설계 단계

## 목적
하위 항목 하나를 구현하기 전에 무엇을 어떻게 만들지 사용자와 합의한다.
설계는 feature 전체가 아니라 **하위 항목 하나** 단위로 이루어진다.

## checklist.md 구조 전제

```
- [ ] feature/3: 내신 · 모의고사 데이터 API
  - [x] grades 서브도큐먼트 스키마 정의
  - [ ] mockExams 서브도큐먼트 스키마 정의   ← 이 항목 하나를 설계
  - [ ] 학기별 평균 등급 자동 계산 로직
  - [ ] ...
```

## 절차

### 1. 대상 항목 선택
`checklist.md`를 읽고 미완료(`- [ ]`) 최상위 feature 항목 중 첫 번째를 찾는다.
그 안에서 미완료 하위 항목 중 **첫 번째 것 하나**를 선택한다.

- checklist.md가 없으면: "프로젝트 루트에 checklist.md가 없어요." 안내 후 종료
- 미완료 항목이 없으면: "모든 항목이 완료됐어요!" 안내 후 종료

선택한 항목을 사용자에게 알린다:
```
[feature명] > [하위 항목명] 을 설계할게요.
```

### 2. feature 브랜치 생성 안내 (신규 feature일 때만)
선택한 하위 항목이 현재 feature의 **첫 번째 하위 항목**(= 위에 완료된 항목이 하나도 없음)일 때만 브랜치 생성을 안내한다.

```
git checkout -b feature/[이슈번호]-[기능명] origin/dev
```

이미 이전 하위 항목이 완료된 적 있다면(= 브랜치가 존재) 이 단계를 건너뛴다.

### 3. 구현 방법 인터뷰
이 하위 항목을 어떻게 구현할지 파악한다. 한 번에 모두 묻지 말고 하나씩 질문한다.

- 어떤 파일을 수정/생성할 것인지
- 어떤 방식으로 구현할 것인지
- 알고 있는 제약 조건이나 요구사항

**프론트엔드 UI 항목일 때는 반드시 `DESIGN.md`를 먼저 읽고** 색상·간격·컴포넌트 규칙·스크린샷을 참고하여 질문하고 설계한다.

질문할 때는 **반드시 선택지 3개를 함께 제시**한다. 마지막 선택지는 항상 "모르겠어 / 제안해줘" 계열로 열어둔다.

### 4. 비판적 검토
파악한 설계 방향에서 발생할 수 있는 문제, 더 나은 방법, 누락된 케이스를 능동적으로 찾는다.

문제가 있으면 무엇이 문제인지, 어떻게 해결하면 좋은지 직접 제안한다.
문제가 없으면 건너뛴다.

### 5. 설계 요약 제시
```
## 설계 요약
- 항목: [하위 항목명]
- 수정/생성할 파일: [파일 목록]
- 구현 방식: [합의된 접근법]
- 주의사항: [엣지 케이스, 제약 조건]
```

### 6. 문서 반영
설계 결정사항 중 프로젝트 전반에 영향을 주는 내용을 `CLAUDE.md`에 반영한다.
반영할 내용이 없으면 건너뛴다.

## 완료 조건
설계 요약 확인 + 문서 반영(또는 건너뜀) 후 사용자가 진행에 동의했을 때 → workflow가 `implement` 스킬을 발동한다
